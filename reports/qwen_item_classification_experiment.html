Qwen Item Classification Experiment Report

This report documents the current local Qwen item-classification experiment for generating flo_items from already extracted receipt line items. Qwen receives merchant and item text only, while category IDs and tags are constrained by the receipt item taxonomy.

Branch: llm-memory Model: Qwen/Qwen3.5-4B Runtime: conda env genai, Apple MPS Report date: 2026-06-28

Executive Summary

5 example scenarios in this report
1 actual Qwen run completed for the report
18 full taxonomy main categories
9 focused main categories for the EDEKA supermarket case

The important design change is merchant-first narrowing. The app first infers a merchant_context, then gives Qwen a focused taxonomy subset. Qwen still classifies each item by item purpose, so a supermarket receipt can contain groceries, household products, personal care items, pet items, deposits, and retail goods in the same transaction.

Pipeline Under Test

1

Receipt Extraction

Gemini extracts merchant, currency, line item names, quantities, and amounts. No transaction category is passed to Qwen.

2

Merchant Context

Local rules infer a merchant type and a candidate range of possible main categories.

3

Focused Taxonomy

The full taxonomy is narrowed to the merchant candidates plus other, reducing prompt size and ambiguity.

4

Qwen + Validation

Qwen outputs JSON. Python validation enforces allowed main categories, subcategories, tags, raw item order, and source.

Current Prompt Contract

  • Qwen input contains merchant, currency, raw_name, quantity, and amount.
  • Qwen does not receive a fabricated transaction category.
  • main_category must be one ID from the allowed focused taxonomy.
  • sub_category must belong to the selected main_category.
  • tags must contain 1 to 4 values from the allowed tag vocabulary.
  • If validation fails, the item falls back to other, other.unknown, and ["unknown"].

Case 1: EDEKA packaged ice cream

Merchant-first narrowing classifies EDIKA/EDIKA as a supermarket. This result is from an actual local Qwen run.

Observed

Input

{
  "merchant_context": {
    "merchant_type": "supermarket",
    "candidate_main_categories": [
      "groceries",
      "household",
      "personal_care",
      "health",
      "pets",
      "children_family",
      "financial_admin",
      "retail_goods"
    ]
  },
  "payload": {
    "merchant": "edika",
    "currency": "EUR",
    "items": [
      {
        "raw_name": "MAGNUM MANDEL",
        "quantity": 1,
        "amount": 2.79
      }
    ]
  }
}

Output

{
  "items": [
    {
      "raw_name": "MAGNUM MANDEL",

      "sub_category": "groceries.snacks_sweets",
      "tags": [
        "dessert",
        "snack",
        "supermarket_item"
      ],
      "confidence": 0.95,
      "classification_source": "qwen"
    }
  ]

Runtime Notes

Observation Impact Recommendation
The EDEKA single-item case completed after the model was loaded, but generation still took about 95 seconds. CLI-style local inference is acceptable for experiments, not for a smooth app flow. Run Qwen as a warm local service process and call it from the app instead of loading it per request.
The multi-item REWE batch was interrupted because generation exceeded the expected wait time. Long prompts plus multi-item output can be slow on the 4B model through this CLI path. Keep merchant-focused taxonomy narrowing, reduce output tokens, and consider classifying item lines in smaller batches.
Python validation protects the saved flo_items fields from out-of-vocabulary model output. The app can safely store stable category IDs even when model text is imperfect. Keep validation as a required post-processing step before writing to localStorage or a future database.

Conclusion

The merchant-first design is the right direction. It gives Qwen a smaller and more relevant decision space while preserving item-level classification. For example, a supermarket merchant should make groceries likely, but it should still allow toothpaste, detergent, pet food, baby supplies, bottle deposits, and occasional retail goods.

The next engineering step is not another prompt tweak. The next step is to turn this demo into a stable local classification service and then connect it to receipt save flow so that flo_expenses remains the transaction index and flo_items becomes the item-level analytics index.
